仅仅导入和仅仅导出#

import type和export type可以称之为仅仅导入申明；仅仅导出申明

import type { Options } from '@/interface'
export type { Options }

当我们不使用仅仅导入；仅仅导出的时候，类型申明照样能使用：

foo.ts
export interface Options {
children: JSX.Element
path: string
}
export function doThing() {
return 'runtime'
}
bar.ts
// 混合导入，照样能使用
import { Options, doThing } from 'foo.ts'
const options: Options = {}
doThing()

使用仅仅导入的区别#

既然能正常使用，那么使用import type又有啥区别呢？

import { MyThing } from './some-module.ts'
export { MyThing }

如上面的代码所示：MyThing到底是一个类型还是一个值，如果单从一个文件来看，我们是无法知道答案的：

解决无法推断是申明还是值#

仅仅导出#

我们可以通过export type来告知编译器，这里代表的是一种类型

import { MyThing } from './some-module.ts'
export type { MyThing }

使用具有上下文功能的编译器#

编译器能够理解整个系统文件比如tsc，tsc编译器能够通过整个系统的上下文分析得出MyThing

tsc src/*.ts

避免风险代码#

开启tsconfig.json：isolatedModules配置，这样可以避免写出这样的风险代码。isolatedModules 详细介绍

{
"compilerOptions": {
"isolatedModules": true
}
}

案例分析#

比如babel,typescript的Node API：transpileModule API，一次只能在一个文件上操作，没有上下文的概念，就会有问题

.
├── src
│ ├── index.ts
│ ├── container.ts
│ ├── module-a.ts
  • index.ts
  • container.ts
  • module-a.ts
import { ModuleA, moduleA } from './module-a'
export { ModuleA, moduleA }
  • index.js
  • container.js
  • module-a.js
import { ModuleA, moduleA } from './module-a'
export { ModuleA, moduleA }
danger

如上图所示：container里面，单从container.ts里面无法分析ModuleA, moduleA是申明还是值，所以就保存了下来：

import { ModuleA, moduleA } from './module-a'
export { ModuleA, moduleA }

但是module-a文件里面ModuleA类型申明，是可以分析出来的，所以babel编译的时候，就删除掉了这个类型申明，只留下了运行时的值moduleA

export const moduleA = 'moduleA'

这个时候container从module-a里面导入ModuleA就报错了
